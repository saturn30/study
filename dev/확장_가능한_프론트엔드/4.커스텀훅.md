# 4. 확장 가능한 프런트엔드 - 커스텀 훅

[https://blog.codeminer42.com/scalable-frontend-4-custom-hooks-to-the-rescue/](https://blog.codeminer42.com/scalable-frontend-4-custom-hooks-to-the-rescue/)

### 규칙

사용자 정의 훅에 대한 몇 가지 중요한 기본 규칙을 설정하자

1. 훅은 사용 중인 UI 라이브러리 또는 프레임워크에서 지시하는 다른 모든 훅 규칙을 따라야 한다.
2. 사용자 정의 훅은 우리가 사용 중인 기술을 누출하거나 드러내서는 안 된다. 훅의 사용자는 redux, 컨텍스트 api, graphQL 등을 사용하고 있다는 사실을 알지 못한다. 그리고 항상 아래에 있는 기술을 드러내지 않는 데이터와 기능이 있는 객체만 노출해야 한다.
3. 훅의 소비자는 상태가 저장된 모양에 대해 많이 알지 않아야 한다. 즉, 상태 변경을 트리거 할 수 있는 훅에 의해 노출되는 기능은 단순히 설정자가 아니라 도메인 명령을 반영해야 한다.
4. 훅은 상태를 너무 많이 덮지 않아야 하며 (가지지 말라는 의미인가 봄), 도메인 지향 부분에 초점을 맞춰야 하며 다른 사용자 정의 훅과 최대한 직교해야 한다. ??
5. 고객은 커스텀 훅에 의해 노출된 작업의 반환에 의존해서는 안 된다.
6. 노출될 수 있는 도메인 논리에서 생성된 데이터를 포함하여 전반적으로 상태에서 파생된 사전 처리된 데이터가 필요한 경우 커스텀 훅을 통해 노출해야 한다.
7. 모든 도메인 로직이 훅에 의해 노출될 필요는 없다. 필요한 경우 도메인 논리를 구성 요소 파일로 직접 가져올 수 있지만 훅은 데이터 및 작업 노출에 중점을 두어야 한다.
